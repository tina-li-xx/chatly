# Concerns Log
- Public blog routes and `sitemap.xml` now skip workspace-backed generated SEO draft reads during the `next build` phase so static generation cannot stall on the database migration/query path. Runtime and ISR requests still merge those generated posts, but a fresh deployment build can ship blog pages and sitemap entries that lag newly published generated drafts until the server has a chance to regenerate them.
- The SEO keyword registry now stores every discovered term and writes SERP snapshots over time, but each refresh cycle only runs canonical SERP snapshots for the top tracked slice of the corpus so the scheduled job stays bounded. Longer-tail discovered keywords still persist in Postgres and can be promoted in later cycles, but they will not all get a fresh canonical snapshot on every refresh.
- Manual regeneration on `/dashboard/publishing` is now one-at-a-time and intentionally conservative: plan-item regeneration only rewrites items that are still in `planned` status. Once a draft already exists, the plan action warns the reviewer to regenerate the draft instead so the topic and article cannot silently drift out of sync.
- The new SEO keyword corpus now persists externally harvested phrases, rankings, and opportunity scores in Postgres, and the planner reads that stored corpus instead of only using transient live queries. The scoring is still heuristic and based on repeated SERP evidence, rank gaps, and competitor density rather than a true search-volume or keyword-difficulty dataset, so the corpus is more durable and useful than the old seed list but still should not be treated like Ahrefs-grade volume intelligence.
- The SEO research layer now prefers a self-hosted `SearXNG` JSON endpoint when `SEARXNG_BASE_URL` is configured, then falls back to Bing RSS and DuckDuckGo HTML. That is better than relying on public scraping alone, but it still depends on either a healthy SearXNG instance or public search formats remaining parseable. If SearXNG is unavailable and the public fallbacks change or tighten bot protections, Chatting falls back to internal heuristics and the live opportunity scoring becomes less trustworthy until the provider strategy is updated.
- Transient Postgres and Redis disconnects are now retried during scheduler window bookkeeping and treated as reconnectable runtime events, but longer provider outages can still defer a scheduler cycle until the next interval and can still drop Redis pub/sub events that occur entirely during the outage window because there is no persisted event backfill.
- Expo CLI currently crashes on this machine's Node v25 runtime when it tries its default free-port scan for the Chatting customer mobile app. The new mobile scripts pin Metro to port `8089` as a workaround, so local startup is stable as long as that explicit port stays available or is updated deliberately.
- Teammate mobile push for the Chatting customer app now uses direct APNs on iOS and direct FCM on Android, with the app bundle/package IDs plus `google-services.json` wired into `apps/mobile/app.json` and Android notifications targeting the `chatting-messages` channel. Background notifications still depend on a real native build and a server runtime with Google Application Default Credentials or service-account impersonation available for FCM; local testing will need `gcloud auth application-default login` or equivalent runtime identity, and Expo Go will not provide the native device-token path this delivery flow expects.
- Runtime brand contracts now hard-cut to `chatting` only. Older `chatly_session` auth cookies, `chatly:*` widget storage, `chatly:widget:mounted` install signals, `chatly_timezone`, and pre-cut Slack/Shopify OAuth state or popup message contracts are no longer migrated or accepted after this rename pass.
- Native iOS push is now wired through APNs, and this workspace already has the required `APPLE_TEAM_ID`, `APPLE_KEY_ID`, and `APPLE_PUSH_KEY_P8` values configured locally. The remaining iOS gap is operational: the mobile app still needs a real native build that requests notification permission, registers for remote notifications, and hands the APNs device token back to Chatting before background delivery can be verified end to end.
- The hosted widget still uses internal `data-chatly-*` selectors, CSS variables, and animation/style identifiers inside `public/widget.js`. Those implementation details are separate from the runtime contract rename and still need a dedicated widget-internals rename if the goal is to remove the old brand string everywhere in hosted widget code.
- Inbound email replies now verify sender addresses against the conversation email plus previously sent visitor-email recipients. Team notification emails intentionally no longer support direct reply-by-email until a separate authenticated team-email reply path exists.
- Team roles are now enforced for invite management and billing routes, but other workspace settings surfaces still assume broad shared access. Full owner/admin/member authorization across all dashboard actions is still incomplete.
- Passive visitor tracking now depends on widget heartbeats plus route-change pings, so browser throttling or missed heartbeats can still make online/offline state lag by up to the polling interval and grace window.
- Dashboard Settings now exposes a single editable team name by renaming the primary site record, but visitor emails still use the specific conversation's `sites.name`. Multi-site workspaces can still show different names across conversations until identity is separated from site records.
- The new SEO sitemap is generated from the current hardcoded public route list plus blog and free-tool data. Any future marketing pages added outside those registries will not reach Google through the sitemap until `app/sitemap.ts` is updated.
- The homepage now leans on product-proof sections and screenshot-style mockups instead of customer metrics/testimonials. Public social proof still needs verified customer references before it can safely return to the landing page.
- Chatting SEO autopilot now resolves its publishing workspace from the internal reviewer account at `tina@usechatting.com`. If that account or its active workspace ownership changes, the public blog merge and the background autopilot will follow that mapping until the publishing-owner lookup is made more explicit.
- Dashboard Publishing now runs live SEO analysis, 30-day plan generation, and article draft generation from current Chatting product/content sources and uses MiniMax when configured, but it still falls back to deterministic source-backed heuristics when the provider is unavailable. The fallback keeps the internal workspace usable, but it will not surface fresh market nuance or draft quality as well as the provider-backed flow.
- The landing-page hero now uses a CSS-timed animated mockup plus a reduced-motion static fallback. If the widget UX or marketing copy changes, both hero variants need to be kept in sync by hand.
- Conversation assignment now includes shared inbox filters plus live automation routing for new conversations, but it is still intentionally lightweight. There are still no per-teammate queue views, true team pools, workload balancing, or SLA tools.
- Member conversation visibility now follows assignment at the repository layer. That keeps non-admin teammates scoped to their own work, but it also means unassigned conversations remain owner/admin-only until someone explicitly assigns them.
- The public privacy policy now spells out Chatting's disclosed mobile data collection and sharing more explicitly for App Review. That policy still needs to stay in sync with the App Store Connect privacy questionnaire and any future changes to account data, message handling, attachments, device-token registration, or subprocessor usage so the next review cycle does not drift back into a disclosure mismatch.
- Saved replies now support shared workspace tags plus inbox insertion, but the library is still intentionally simple. There are still no variables, categories, usage analytics, or ordering tools for larger teams.
- Settings sections still mix padded form cards with a few list-style cards built from separate primitives, so layout consistency can drift when a section switches between loading, empty, and populated states.
- CRM-lite now lives inside the inbox sidebar and visitors drawer, but the inbox view still only has aggregate visitor history. Deeplinkable multi-conversation history is available from the visitors drawer, not directly inside each inbox thread yet.
- A public identify endpoint now exists at `/api/public/identify`, but the embeddable widget API surface still does not expose a first-class `Chatting('identify', ...)` helper in the hosted script. Developers can call the route directly, but the widget-side DX is incomplete.
- AI assist is now available in the inbox composer for manual summary, rewrite, reply, and tag suggestions, but it still depends on MiniMax being configured. Suggestions are generated from the current thread only and are not grounded against a help center or policy source yet.
- The full AI Assist activity page now uses a lighter cursor feed and actor index for `Load more`, but filter changes still navigate through a full server render and repopulate the member list on each change instead of using a fully client-cached filter model.
- The inbox now uses a lighter left-panel summary query, and the active thread now defers visitor-history sidebar data into a second background request. If `/dashboard/inbox?id=...` still feels slow, the next deeper pass is trimming the full message-history load itself rather than the list or visitor-activity aggregate.
- Help center lite now supports workspace-owned public articles under `/help/[siteId]`, but it is still intentionally minimal. Articles are plain-text only, shared across all sites in the workspace, and are not yet included in sitemap/search or surfaced directly inside the widget/inbox reply flow.
- The new Automation settings page now persists workspace-level automation rules and syncs overlapping offline/widget copy. FAQ suggestions now gate the first team notification until the visitor explicitly asks for a human or sends a follow-up message, and the shown FAQ snapshot now survives widget reloads. Proactive prompts now run live in the widget with per-session matching, delay, and auto-open behavior, but the older Widget Settings `autoOpenPaths` control still exists as a second proactive entry point and can confuse teams about which surface owns proactive outreach.
- Automation routing now supports drag-and-drop ordering plus additional rule shapes like `page_url_starts_with` and `custom_field_equals` in settings, but assignment targets are still limited to round-robin or an individual teammate. True team targets and richer picker-backed inputs like country/custom-field registries still need follow-through work.
- Auto-routing now runs live for new incoming conversations across page URL, first message, referrer, visitor country, visitor tags, and custom fields. The remaining gap is profile sourcing: `visitor_tag` and `custom_field_equals` rules only match when the widget or public API is given those profile values, and there is still no dedicated dashboard CRM/editor to manage visitor tags/custom fields after a visitor is created.
- Settings section navigation now uses real section URLs and section-scoped server loads, but unsaved-change protection in settings still relies on the browser's unload prompt rather than an in-app route guard or confirmation modal.
- Slack settings now encrypt stored credentials at rest before writing them into `workspace_integrations.credentials_json`, and the dashboard auto-upgrades any legacy plaintext Slack credential row the next time it is loaded. If `INTEGRATIONS_ENCRYPTION_KEY` is unset, the encryption helper falls back to `AUTH_SECRET`, so deployments that want stricter key separation should still provide a dedicated integrations key.
- Slack now posts signed new-conversation notifications and accepts signed Slack Events API thread replies back into the normal team-reply delivery path, but the other Slack notification toggles in settings (`assigned to me`, `resolved`, `all new messages`) are still UI-only.
- Slack thread replies currently land in Chatting as generic team messages. The workspace can reply from Slack, but Slack user IDs are not yet mapped back to individual Chatting teammates for author attribution in the inbox.
- Slack settings still use the current static channel picker. Delivery upgrades to the real Slack channel ID after the first successful post, but the settings UI still does not browse the connected workspace's actual channel list, and Slack event retries are not yet deduplicated across partial failures.
- Webhook settings still rely on browser-local demo state in the dashboard UI and do not yet share real backend persistence, teammate sync, or live provider/API delivery.
- Zapier now has live workspace API keys, webhook subscriptions, `/api/zapier/...` trigger/action endpoints, background-queued event delivery, and idempotent incoming action routes. The main remaining operational gap is external to this repo: Chatting's Zapier developer-platform app still has to be configured separately against these endpoints.
- Shopify now connects through a real OAuth start/callback flow and live inbox lookup, but the sidebar still depends on an exact visitor-email match against Shopify customers. Stores that need older order history or protected customer data on live shops may still need Shopify approval beyond the current `read_customers` and `read_orders` scopes.
- The trial-expired marketing email now renders Growth pricing as shared tier rows instead of one wrapped sentence. If Growth pricing adds more tiers or promo copy, the shared pricing breakdown helper and the email layout still need to be updated together so the panel stays tidy.
