# Concerns Log
- The app build still warns that `metadataBase` is unset in metadata exports, so social Open Graph/Twitter asset URLs fall back to `http://localhost:3000` during production builds unless those routes set an explicit base URL.
- At least one app route still uses the Edge runtime, which makes that page opt out of static generation during builds. That behavior may be intentional, but it remains a production-performance tradeoff to review.
- Health scoring currently uses the owner workspace's `auth_sessions` history as the login-frequency signal. Accepted teammate memberships now exist, but the score is still owner-weighted rather than truly team-wide.
- Signup, site creation, and referral attribution are still separate database writes rather than a single transaction. Invalid referral codes are validated before account creation now, but an infrastructure failure after the user row is created could still leave an account without its intended referral attribution.
- Newsletter signups now persist locally and sync to Resend contacts, but dedicated newsletter list routing is only automatic when `RESEND_NEWSLETTER_SEGMENT_ID` or the legacy `RESEND_NEWSLETTER_AUDIENCE_ID` is configured. Without one of those env vars, subscribers sync to Resend without landing in a dedicated newsletter segment automatically.
- The newsletter flow still does not expose a subscriber-facing unsubscribe/preferences route in the app. Contact opt-out and list hygiene currently depend on Resend-side management rather than an in-product settings page.
- The shared Chatting email design system now covers transactional, notification, visitor, and lifecycle templates, but only the flows that already have natural product triggers are wired live. Trial-ending reminders, trial-expired nudges, product updates, and verification emails still have render/send modules ready without scheduler or campaign orchestration yet.
- Conversation-note mention emails now send from real `@handle` matches, but the parser is still plain-text and single-token. Ambiguous handles like two active teammates both matching `@tina` are skipped rather than resolved through autocomplete or structured mentions.
- Weekly performance emails now send from the live analytics dataset, but their highlight copy is limited to the metrics the current analytics model can compute. Richer weekly insights like teammate-first-live-chat milestones or true conversion analysis still are not derived yet.
- Email clients still have uneven support for modern layout primitives in HTML email. The team invite card now uses table-first markup after a client rendering regression, but other templates still rely on `div`-heavy sections that should keep getting spot-checked in real inboxes.
- Settings template test sends now use optimistic success notices so the footer CTA stays stable, but a failed network request can still briefly show success before the error notice replaces it.
- `src/lib/email-templates.ts` is now well over the repo's 200-line code-file limit. The recent template copy and rendering fixes stayed surgical, but that template registry/renderer still needs to be split before any commit.
- `app/dashboard/dashboard-settings-profile-section.tsx` is still over the repo's 200-line code-file limit. The settings page split removed the main page bottleneck, but the profile section still needs to be broken up before any commit.
- `app/dashboard/settings-email-template-ui.tsx` is still well over the repo's 200-line code-file limit. The modal layout fix stayed surgical, but that editor/preview UI still needs to be split before any commit.
- The new hosted `/conversation/<token>` resume page currently supports text replies only. Existing threads still display attachment links, but uploading new attachments from the hosted page is not wired yet.
- Settings preview/test emails now use the hosted conversation-link shape, but they still rely on preview placeholder URLs rather than real signed conversation identities because the template editor has no live visitor thread context.
- Password reset now uses real one-time email tokens, but email verification is still not a required account state in the product. The verification template exists, but signup/login do not enforce verified addresses yet.
- Team invite links now support real acceptance and workspace membership, but the product still assumes one workspace context per user. Accounts that already own a workspace are blocked from joining another one until multi-workspace support exists.
- Team roles are now enforced for invite management and billing routes, but other workspace settings surfaces still assume broad shared access. Full owner/admin/member authorization across all dashboard actions is still incomplete.
- Visitor-facing email legal footers still link only to Chatting's privacy policy because there is no recipient-aware unsubscribe/preferences endpoint in the app yet.
- Marketing copy can still drift ahead of shipped product capabilities. Known mobile-app mentions have been removed from the landing and blog surfaces, but future copy changes still need product review so unsupported features do not get promised again.
- Passive visitor tracking now depends on widget heartbeats plus route-change pings, so browser throttling or missed heartbeats can still make online/offline state lag by up to the polling interval and grace window.
- `app/dashboard/dashboard-settings-shared.tsx` and `app/dashboard/dashboard-widget-settings-shared.tsx` are still over the repo's 200-line code-file limit. The new pricing rewrite touched shared billing messaging there, but those shared settings helpers still need to be split before any commit.
- `app/dashboard/dashboard-thread-detail.tsx` is still well over the repo's 200-line code-file limit. This cleanup deduplicated its visitor identity handling, but the main thread-detail view still needs to be split before any commit.
- The dashboard billing slider is a pricing preview only. Checkout and Stripe seat synchronization still bill from the workspace's actual active team-member count rather than any hypothetical seat count the user previews in the UI.
- Teams at `50+` members now hit a custom-quote state in dashboard billing, but the product still does not have a dedicated in-app sales/contact handoff route. The UI blocks automatic checkout for those teams and explains the limit, but the follow-through path still needs a real lead-capture or contact workflow.
- The shared visitor note is available from the conversation sidebar, but the section label is still just `Notes`. Even with the helper copy softened, that surface can still read as conversation-local unless the label itself is clarified.
- Widget availability now reflects the freshest recent presence across the workspace owner and active teammates, but it is still only a presence-based proxy for coverage. A teammate can look "online" from recent activity even if nobody is intentionally staffing chat at that moment.
- `src/lib/data/sites.ts`, `src/lib/repositories/sites-repository.ts`, and `src/lib/repositories/shared-repository.ts` are still over the repo's 200-line code-file limit. The offline-copy settings work stayed surgical inside those modules, but they still need follow-up splits before any commit.
