# Concerns Log
- Inbound email replies now verify sender addresses against the conversation email plus previously sent visitor-email recipients. Team notification emails intentionally no longer support direct reply-by-email until a separate authenticated team-email reply path exists.
- The Reddit reply generator prompt at `digital_marketing/chatting-reddit-reply-generator.md` is currently untracked in git, so prompt rule updates there will stay local until that file is explicitly added to version control.
- Team roles are now enforced for invite management and billing routes, but other workspace settings surfaces still assume broad shared access. Full owner/admin/member authorization across all dashboard actions is still incomplete.
- Passive visitor tracking now depends on widget heartbeats plus route-change pings, so browser throttling or missed heartbeats can still make online/offline state lag by up to the polling interval and grace window.
- Dashboard Settings now exposes a single editable team name by renaming the primary site record, but visitor emails still use the specific conversation's `sites.name`. Multi-site workspaces can still show different names across conversations until identity is separated from site records.
- The new SEO sitemap is generated from the current hardcoded public route list plus blog and free-tool data. Any future marketing pages added outside those registries will not reach Google through the sitemap until `app/sitemap.ts` is updated.
- The homepage now leans on product-proof sections and screenshot-style mockups instead of customer metrics/testimonials. Public social proof still needs verified customer references before it can safely return to the landing page.
- Conversation assignment now includes shared inbox filters plus live automation routing for new conversations, but it is still intentionally lightweight. There are still no per-teammate queue views, true team pools, workload balancing, or SLA tools.
- Saved replies now support shared workspace tags plus inbox insertion, but the library is still intentionally simple. There are still no variables, categories, usage analytics, or ordering tools for larger teams.
- Settings sections still mix padded form cards with a few list-style cards built from separate primitives, so layout consistency can drift when a section switches between loading, empty, and populated states.
- CRM-lite now lives inside the inbox sidebar and visitors drawer, but the inbox view still only has aggregate visitor history. Deeplinkable multi-conversation history is available from the visitors drawer, not directly inside each inbox thread yet.
- A public identify endpoint now exists at `/api/public/identify`, but the embeddable widget API surface still does not expose a first-class `Chatly('identify', ...)` helper in the hosted script. Developers can call the route directly, but the widget-side DX is incomplete.
- AI assist is now available in the inbox composer for manual summary, rewrite, reply, and tag suggestions, but it still depends on MiniMax being configured. Suggestions are generated from the current thread only and are not grounded against a help center or policy source yet.
- The full AI Assist activity page now uses a lighter cursor feed and actor index for `Load more`, but filter changes still navigate through a full server render and repopulate the member list on each change instead of using a fully client-cached filter model.
- Help center lite now supports workspace-owned public articles under `/help/[siteId]`, but it is still intentionally minimal. Articles are plain-text only, shared across all sites in the workspace, and are not yet included in sitemap/search or surfaced directly inside the widget/inbox reply flow.
- The new Automation settings page now persists workspace-level automation rules and syncs overlapping offline/widget copy. FAQ suggestions now gate the first team notification until the visitor explicitly asks for a human or sends a follow-up message, and the shown FAQ snapshot now survives widget reloads. Proactive prompts now run live in the widget with per-session matching, delay, and auto-open behavior, but the older Widget Settings `autoOpenPaths` control still exists as a second proactive entry point and can confuse teams about which surface owns proactive outreach.
- Automation routing now supports drag-and-drop ordering plus additional rule shapes like `page_url_starts_with` and `custom_field_equals` in settings, but assignment targets are still limited to round-robin or an individual teammate. True team targets and richer picker-backed inputs like country/custom-field registries still need follow-through work.
- Auto-routing now runs live for new incoming conversations across page URL, first message, referrer, visitor country, visitor tags, and custom fields. The remaining gap is profile sourcing: `visitor_tag` and `custom_field_equals` rules only match when the widget or public API is given those profile values, and there is still no dedicated dashboard CRM/editor to manage visitor tags/custom fields after a visitor is created.
- Settings section navigation now uses real section URLs and section-scoped server loads, but unsaved-change protection in settings still relies on the browser's unload prompt rather than an in-app route guard or confirmation modal.
